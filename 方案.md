# Android 16KB Native Library 检测与依赖定位工具

> 目标：
>
> * 分析 **APK / AAB** 中不符合 **16KB page size** 要求的 `.so`
> * 精确定位该 `.so` **是由哪个依赖（AAR / module / 本地 jniLibs）引入**
> * 以 **Gradle 插件 + CLI + Android Studio 插件** 的形式提供，便于 CI/CD 与日常开发使用

---

## 1. 背景与问题定义

随着 Android 15 / 新设备对 **16KB page size** 的支持，包含 native code 的应用如果仍使用 **4KB 对齐的 `.so`**，可能在运行时出现 **加载失败 / 安装失败 / Google Play 兼容性问题**。

现实痛点：

* APK / AAB 中出现不合格 `.so`
* `.so` 往往来自 **第三方 SDK**
* 现有工具只能发现问题，**无法定位来源**

本工具要解决的核心问题是：

> **这个不合格的 `.so` 是谁带进来的？**

---

## 2. 总体架构设计

采用 **三层架构**，降低 AGP / IDE 变动风险：

```

+-------------------------+
```

## 3. 检测规则定义（16KB 不合格判定）

### 3.1 ELF Program Header 对齐（核心）

针对每个 `.so`，解析 ELF **Program Headers**，检查所有 `PT_LOAD` 段：

* `p_align >= 16384 (0x4000)`
* `p_vaddr % 16384 == p_offset % 16384`

> 任一 `PT_LOAD` 不满足即判定 **FAIL**

可选增强：

* `p_offset % 16384 == 0`（作为 warning）

---

### 3.2 ZIP 内数据偏移对齐（APK/AAB 层）

即使 ELF 合格，如果 `.so` 在 ZIP 中的 **data offset** 不满足 16KB 对齐，也可能无法直接 mmap。

检查规则：

```
dataOffset % 16384 == 0
```

其中：

```
dataOffset = localHeaderOffset + 30
           + fileNameLength
           + extraFieldLength
```

---

### 3.3 `.so` 压缩方式

检查 ZIP entry 的 compression method：

* `STORED (0)`：推荐
* `DEFLATE`：记录为问题（warning / error 可配置）

---

## 4. APK / AAB 扫描方案

### 4.1 APK

* APK 本质为 ZIP
* 扫描路径：

```
lib/<abi>/*.so
```

对每个 `.so`：

* 读取 bytes（或写入临时文件）
* 计算 `sha256`
* 执行 ELF 检测
* 执行 ZIP 对齐 / 压缩检测

---

### 4.2 AAB

AAB 为模块化 ZIP：

```
base/lib/<abi>/*.so
<feature>/lib/<abi>/*.so
```

扫描规则：

```
^(.*?)/lib/([^/]+)/(.+\.so)$
```

其余流程同 APK。

---

## 5. 依赖归因（核心价值）

### 5.1 总体思路

构建 **hash → 来源依赖** 映射表：

1. 构建期收集所有可能来源的 `.so`
2. 计算 `sha256`
3. 记录来源（GAV / project path）
4. 扫描最终 APK/AAB 的 `.so`
5. 用 hash 精确反查来源

---

### 5.2 可能的 `.so` 来源

| 来源                  | 示例                  | 归因方式           |
| ------------------- | ------------------- | -------------- |
| Maven AAR           | `com.xxx:lib:1.2.3` | GAV            |
| Project module      | `:weexcontainer`    | project path   |
| 本地 jniLibs          | `src/main/jniLibs`  | project + path |
| externalNativeBuild | CMake / ndk-build   | project + task |
| files() AAR         | `libs/xxx.aar`      | FileCollection |

---

### 5.3 AAR 依赖扫描（推荐方案）

使用 **ArtifactView + Transform**：

* 输入：AAR
* Transform 行为：

  * 解压 AAR
  * 遍历 `jni/<abi>/*.so`
  * 计算 `sha256`
  * 输出 mapping：

```json
{
  "sha256": "...",
  "abi": "arm64-v8a",
  "soName": "libfoo.so",
  "origin": {
    "type": "maven",
    "group": "com.xxx",
    "name": "sdk",
    "version": "1.2.3"
  }
}
```

Transform 优点：

* 支持缓存
* 支持增量
* 性能稳定

---

### 5.4 本地 module / jniLibs

* 扫描参与当前 variant 的 `jniLibs` 目录：

```
src/main/jniLibs
src/<flavor>/jniLibs
src/<variant>/jniLibs
```

* externalNativeBuild：

  * 从 variant 的 native build task outputs 获取 `.so`

归因：

```
Origin.Project(path=":app", source="jniLibs")
```

## 6. Gradle 插件设计

### 6.1 Extension

```kotlin
check16k {
  pageSize = 16384
  strict = true
  checkZipAlignment = true
  checkCompressed = true
  inferOrigin = true
  reportDir = file("$buildDir/reports/16k")
  formats = ["json", "html", "sarif"]
}
```

---

### 6.2 Tasks

为每个 variant 注册：

* `check16kDebug`
* `check16kRelease`

任务职责：

1. 收集依赖 `.so` hash 映射
2. 定位 APK / AAB 输出路径
3. 扫描产物并检测
4. 生成报告
5. strict=true 时 fail 构建

---

```json
  "variant": "release",
  "summary": {
    "totalSo": 132,
    "fail": 2,
    "warn": 6

      "sha256": "...",

        { "type": "ELF_ALIGN", "detail": "p_align=4096" },

        {
          "type": "maven",
          "gav": "com.xxx:sdk:1.2.3",
          "confidence": 1.0

      ],

        "升级 SDK 到支持 16KB 的版本",
        "或移除该依赖"
      ]
    }
  ]
}
```

## 8. Android Studio 插件方案

### 8.1 功能

* 菜单：`Tools → 16KB Checker`
* 运行对应 `check16k<Variant>` task
* 解析 JSON 报告
* ToolWindow 树状展示结果

### 8.2 交互

```
Variant
 └─ ABI
    └─ libfoo.so
       ├─ Issues
       ├─ Origin: com.xxx:sdk:1.2.3
       └─ Suggest
```

点击 Origin：

* Project 依赖：打开对应 module `build.gradle`
* Maven 依赖：全局搜索 `group:name`

## 9. ELF 解析实现要点

* 只需解析 ELF Header + Program Headers
* 不需要 Section Headers
* 支持 32/64 位 + endian 判断

流程：

1. 读取 `e_ident`
2. 解析 `e_phoff`, `e_phnum`, `e_phentsize`
3. 遍历 `PT_LOAD`
4. 执行对齐校验

---

## 10. 常见坑与边界

* 同名 `.so`（如 `libc++_shared.so`）→ 必须用 hash
* AAB 多 module → base + feature 都要扫
* `files("libs/*.aar")` → 只能 best-effort 归因
* 第三方闭源 `.so` 无法修复，只能提示
